# Use CDN-cached redirects and HTTP analytics

Package download redirects should be cacheable `302` responses with a long shared TTL rather than per-request Worker accounting events. Redirect rules remain static lookup data; the project should not introduce a separate rules cache as the primary optimization. The deployable tree publishes `/redirect-rules/snapshot.json.zst`, a zstd-compressed UTF-8 JSON redirect snapshot for release-time comparison, while the Worker reads per-entry redirect shards and stores the generated `302` response in Cloudflare's Cache API. The project should not use Cloudflare Pages `_redirects` for package redirects, because that would turn every package artifact path into a static redirect rule and reintroduce Pages redirect-count limits. The snapshot has a versioned top-level object with a `redirects` map from full package download paths to upstream package file URLs. The publish workflow should fetch the previously deployed snapshot from Cloudflare and actively purge cached package download URLs whose redirect target changed, so cached `302` responses can stay long-lived without leaving known changed paths stale. New versioned package paths usually do not need purging because they have no prior cached response, but active purge protects stable virtual filenames such as `fastfetch-linux-amd64.deb` when their upstream target changes. Download statistics are derived from Cloudflare HTTP request analytics over `GET /pool/*` package download paths, because Worker runtime telemetry is not the public download-count source of truth. `HEAD` requests are excluded from public download counts because they can represent probes, checks, or cache validation rather than a package download.
