# Use CDN-cached redirects and HTTP analytics

Package download redirects should be cacheable CDN `302` responses with a long edge TTL rather than per-request Worker accounting events. Redirect rules remain static lookup data; the project should not introduce a separate rules cache as the primary optimization. The deployable tree publishes `/redirect-rules/snapshot.json.zst`, a zstd-compressed UTF-8 JSON redirect snapshot for release-time comparison, while the Worker reads per-entry redirect shards. The snapshot has a versioned top-level object with a `redirects` map from full package download paths to upstream package file URLs. The publish workflow should fetch the previously deployed snapshot from Cloudflare and actively purge cached package download URLs whose redirect target changed, so cached `302` responses can stay long-lived without leaving known changed paths stale. New versioned package paths usually do not need purging because they have no prior cached response, but active purge protects stable virtual filenames such as `fastfetch-linux-amd64.deb` when their upstream target changes. Download statistics are derived from Cloudflare HTTP request analytics over `GET /pool/*` package download paths, because Worker invocation counts become distorted once redirect responses are cached and should not be the source of truth for public download counts. `HEAD` requests are excluded from public download counts because they can represent probes, checks, or cache validation rather than a package download.
